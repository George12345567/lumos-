# Pricing Request Flow Fix Report — Lumos

## 1. Summary

Fixed the data-flow gap that caused pricing requests submitted by logged-in
clients to land in the admin dashboard as "Unnamed" with no contact info and no
discount code visible. Logged-in submissions now persist a full contact snapshot
(name, phone, email, company) and the active discount code/breakdown alongside
`client_id`, so every admin card has a real name and the discount is auditable.
Status changes already propagated through `status_history`; clients can now
also see their submitted pricing requests with live status inside their profile
Projects tab.

## 2. Root Cause

**Why requests arrived as "Unnamed"** — `RequestCard` in
`src/features/admin/sections/RequestsSection.tsx` rendered the row name as
`request.guest_name || request.company_name || 'Unnamed'`. The pricing modal
only populated those fields for **guest** submissions. For logged-in clients it
sent `client_id` only, so both `guest_name` and `company_name` were `null` on
the row and the admin fell back to the placeholder. The linked client record
was never consulted.

**Why the discount code was missing** — `submitPricingRequest` in
`src/services/pricingRequestService.ts` already accepted `appliedPromoCode` and
`discountBreakdown`, but **none of the three submission paths in the modal**
(`handleSend`, `handleSaveToDashboard`, `handleConfirm`) actually passed them.
The modal's `appliedPromo` state was rendered in the price summary but dropped
on submit, so the row had `applied_promo_code = null` and a zeroed
`discount_breakdown`.

**Why status changes did not reflect for clients** — Status updates were
already persisted correctly (`updatePricingRequestStatus` writes `status` plus
appends to `status_history` and notifies the client). The missing piece was a
view: the client profile only rendered `orders`, never `pricing_requests`, so
clients never saw the request that the admin was working on.

## 5. Logged-in Client Flow

When `isAuthenticated && !isAdmin && client?.id`, `buildPricingPayload()`
produces:

```ts
{
  clientId: client.id,
  guestContact: {
    name: client.full_contact_name || client.display_name || client.username,
    phone: client.phone || client.phone_number,
    email: client.email,
  },
  companyName: client.company_name,
  appliedPromoCode: appliedPromo?.code,
  discountBreakdown: { base_discount, promo_discount, reward_discount, total_discount_percent },
  // …pricing fields
}
```

The service writes the contact snapshot into `guest_name/guest_phone/guest_email`
(those columns hold any contact snapshot regardless of guest vs. client) and
the company into `company_name`. `client_id` is what flags this as a real
client request — that drives the green **Client** badge in the admin and the
filter `client.id = pricing_requests.client_id` in the profile Projects tab.

If the auth client profile is missing fields, the modal still uses whatever it
has plus what the user typed in the contact step — the request is never blocked
by missing optional fields.

## 6. Guest Flow

When the user is unauthenticated, `buildPricingPayload()` reads from the
modal's `contact` state (collected in the Details step):

```ts
{
  // no clientId
  guestContact: { name, phone, email },
  companyName: undefined,        // guest does not provide company in this build
  appliedPromoCode, discountBreakdown,
  locationUrl: window.location.href,
}
```

`request_source` is set to `guest_pricing_modal` in the service. The admin
card shows a slate **Guest** badge. RLS allows the anon insert.

## 7. Discount Code Handling

**Validation** — Unchanged. `validateDiscountCode(code)` in
`src/services/discountService.ts` checks `is_active`, `valid_from`,
`valid_until`, and `usage_limit` against the `discount_codes` table.
The modal's promo input runs this on Apply.

**Persistence** — `buildPricingPayload()` always includes the active
`appliedPromoCode` and a derived `discountBreakdown` so the snapshot is saved
on the row at submission time. The service normalizes the code to uppercase
before insert.

**Admin display** — The request card now shows the promo code (monospace,
emerald) and the total discount amount in the request currency. The drawer's
Pricing tab already had the editable `applied_promo_code` field; that
continues to work and is now consistently populated.

**Client display** — The new Pricing Requests panel shows the promo code +
discount amount on each request card.

## 8. Admin Dashboard Requests

Added to `RequestCard`:

- **Client/Guest badge** — driven by `client_id` presence.
- **Promo code chip** — monospaced uppercase code when present.
- **Discount amount line** — `−amount currency` summed from `discount_breakdown`.
- **Linked-profile hint** — small caption "From profile: email" when the row's
  contact snapshot is empty but `client_id` resolves to a client (legacy rows).

The drawer title and the search filter now use the same
`resolveRequestDisplayName(request, clients)` helper. Existing quick actions
(WhatsApp, Email, Open client, status dropdown) are preserved untouched.

## 9. Status Tracking

`updatePricingRequestStatus` (already in place) writes:

- `status` (validated by `sanitizePricingRequestUpdate`)
- `status_history += { status, changed_at, changed_by, note }`
- `reviewed_at` when transitioning to `reviewing/approved/rejected`
- `updated_at`

It then writes an audit log entry and (when `client_id` is set) a client
notification with localized title/message. The History tab in the admin drawer
renders the timeline. The new client Projects tab surfaces the latest status
on each request card, so the next time the client opens `/profile?tab=projects`
the admin's status change is visible immediately.

## 10. Client Profile Tracking

Logged-in clients see their pricing requests in
`/profile` → Projects tab via the new `ClientRequestsPanel`:

- Invoice number chip
- Status badge (localized — New / In Review / Approved / Converted / Needs revision)
- Package name (or "Custom plan" / "Pricing request")
- Submitted timestamp
- Estimated total + currency
- Discount amount + applied promo code (if any)
- Notes (collapsible block when present)

Refresh button re-runs `getClientPricingRequests(client.id)` which RLS scopes
to `client_id = auth.uid()`.

## 11. RLS / Security

No RLS changes. Verified the existing policy set is sufficient:

- `pricing_requests` — anon/auth INSERT (modal), auth SELECT own
  (`client_id = auth.uid()`), admin full via `is_admin()`. No public SELECT,
  no broad `using(true)`.
- `discount_codes` — admin-managed; client validation via `validateDiscountCode`
  reads only by `code` + `is_active`. Code redemption (usage_count increment)
  remains best-effort and **non-atomic** — see TODOs.
- No service-role keys touched. No new env vars. `dist/` was scanned and
  contains no `service_role`, `SUPABASE_SERVICE`, `SERVICE_ROLE`, `VITE_AI`,
  `sk_live`, `sk_test`, or `sb_secret` strings.

Other guarantees verified by inspection of the existing code:

- Client cannot edit `status`, `priority`, `assigned_to`, `admin_notes`,
  `estimated_total`. Only the admin sanitizer in `adminDashboardService` writes
  those, and that path requires `is_admin()`.
- Client cannot read other clients' rows because `getClientPricingRequests`
  filters by `client_id` and RLS double-checks at the row level.

## 14. Remaining TODOs

These are known gaps not addressed by this change. They are explicitly called
out so the next pass can plan them:

1. **Atomic discount code redemption** — `applyDiscountCode` in
   `src/services/discountService.ts` does a non-atomic
   read-then-update of `usage_count` from the browser. Two concurrent
   submissions can over-redeem a code with `usage_limit = 1`. The correct fix
   is a single Postgres function (security-definer) that validates the code,
   increments `usage_count`, and inserts the request in one transaction, with
   a UNIQUE-constrained "redemption" log table for hard idempotency.
   **Not implemented** here — would require a new migration and an Edge
   Function or RPC; flagged as the next discount-related task.

2. **Guest → client conversion** — when a guest later signs up with the same
   email/phone, their previous `pricing_requests` rows are not yet
   back-linked to the new `client_id`. **Not implemented.** Would require an
   admin-run linking action or a trigger that reconciles on signup.

3. **Server-side discount validation at insert time** — currently the modal
   applies the discount client-side before submit; a malicious client could
   strip or fake the discount fields. The submission path trusts what the
   modal sends. The atomic-redemption RPC above would also close this gap.

4. **Admin notification on new requests** — `submitPricingRequest` currently
   creates a *client*-side notification (so the client sees their own request
   in their bell), but does not fan out to admin/team. Adding an admin
   notification (or a Slack/email webhook) is a separate task.

5. **Cover image / company snapshot for guest** — the guest flow does not
   collect `company_name`. If the business wants company on guest cards too,
   the contact step needs a company input.

6. **`npm audit` not run** — no dependencies changed in this PR; still worth
   running periodically.
